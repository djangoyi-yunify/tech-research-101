# 任务清单：[功能名称]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档
**前置条件**：plan.md（必需），spec.md（用户故事必需），research.md，data-model.md，contracts/

- **单一项目**：仓库根目录的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`

- 以下路径假设为单一项目——根据 plan.md 的结构进行调整

## 依赖关系与执行顺序

### 阶段依赖

- **初始化（阶段 1）**：无依赖——可以立即开始
- **基础层（阶段 2）**：依赖于初始化完成——阻塞所有用户故事
- **用户故事（阶段 3+）**：都依赖于基础层完成
  - 用户故事可以并行进行（如果有人力）
  - 或按优先级顺序依次进行（P1 → P2 → P3）
- **优化（最后阶段）**：依赖于所有期望的用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：可以在基础层（阶段 2）之后开始——不依赖其他故事
- **用户故事 2（P2）**：可以在基础层（阶段 2）之后开始——可以与 US1 集成但应可独立测试
- **用户故事 3（P3）**：可以在基础层（阶段 2）之后开始——可以与 US1/US2 集成但应可独立测试

### 每个用户故事内部

- 测试（如包含）必须在实现之前编写并确保失败
- 模型在服务之前
- 服务在端点之前
- 核心实现在集成之前
- 故事完成后再进入下一个优先级

### 并行机会

- 所有标记为 [P] 的初始化任务可以并行运行
- 所有标记为 [P] 的基础层任务可以并行运行（在阶段 2 内）
- 一旦基础层阶段完成，所有用户故事可以并行开始（如果团队容量允许）
- 同一用户故事的所有测试标记为 [P] 可以并行运行
- 同一故事内标记为 [P] 的模型可以并行运行
- 不同团队成员可以并行处理不同的用户故事

## 实现策略

### MVP 优先（仅限用户故事 1）

1. 完成阶段 1：初始化
2. 完成阶段 2：基础部分（关键 - 阻塞所有故事）
3. 完成阶段 3：用户故事 1
4. **暂停并验证**：独立测试用户故事 1
5. 如果准备就绪则部署/演示

### 增量交付

1. 完成初始化 + 基础部分 → 基础就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事都增加价值且不会破坏已有功能

### 并行团队策略

多个开发人员协同时：

1. 团队共同完成初始化 + 基础部分
2. 基础部分完成后：
   - 开发人员 A：用户故事 1
   - 开发人员 B：用户故事 2
   - 开发人员 C：用户故事 3
3. 各故事独立完成并集成

## 注意事项

- [P] 任务 = 不同文件，无依赖关系
- [Story] 标签将任务映射到特定用户故事以便追溯
- 每个用户故事都应能独立完成和测试
- 在实现前验证测试是否失败
- 每完成一个任务或逻辑分组后提交
- 在任何检查点暂停以独立验证故事
- 避免：模糊任务、同文件冲突、跨故事依赖破坏独立性
