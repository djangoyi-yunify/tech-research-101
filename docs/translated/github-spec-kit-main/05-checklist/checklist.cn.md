## 检查清单目的："需求编写的单元测试"

**核心概念**：检查清单是**需求编写的单元测试**——它们验证特定领域需求的质量、清晰度和完整性。

**不是用于验证/测试**：

- ❌ 不是"验证按钮点击是否正确"
- ❌ 不是"测试错误处理是否有效"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否符合规范

**用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层级要求？" (完整性)
- ✅ "'突出显示'是否用具体的尺寸/位置进行了量化？" (清晰度)
- ✅ "所有交互元素的悬停状态要求是否一致？" (一致性)
- ✅ "是否为键盘导航定义了无障碍要求？" (覆盖率)
- ✅ "规范是否定义了 logo 图片加载失败时的处理方式？" (边界情况)

**比喻**：如果你的规范是用英语编写的代码，那么检查清单就是它的单元测试套件。你在测试需求是否编写良好、完整、无歧义、可以开始实现——而不是测试实现是否有效。

```text
$ARGUMENTS
```

## 执行前检查

**检查扩展钩子（生成检查清单之前）**：
- 检查项目根目录是否存在 `.specify/extensions.yml`
- 如果存在，读取文件并查找 `hooks.before_checklist` 键下的条目
- 如果 YAML 无法解析或无效，静默跳过钩子检查并正常继续
- 过滤掉 `enabled` 明确设为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为已启用
- 对于每个剩余的钩子，**不要**尝试解释或评估钩子 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或为 null/空，则视该钩子为可执行
  - 如果钩子定义了非空 `condition`，则跳过该钩子，将条件评估留给 HookExecutor 实现
- 对于每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：

```
## Extension Hooks

    **Optional Pre-Hook**: {extension}
    Command: `/{command}`
    Description: {description}

    Prompt: {prompt}
    To execute: `/{command}`
```

- **强制钩子**（`optional: false`）：

```
## Extension Hooks

    **Automatic Pre-Hook**: {extension}
    Executing: `/{command}`
    EXECUTE_COMMAND: {command}

    Wait for the result of the hook command before proceeding to the Execution Steps.
```

- 如果没有注册任何 hook 或 `.specify/extensions.yml` 不存在，静默跳过

## 执行步骤

1. **初始化**: 从仓库根目录运行 `{SCRIPT}` 并解析 JSON 获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须为绝对路径。
   - 对于单引号参数如 "I'm Groot"，使用转义语法：例如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **澄清意图（动态）**: 推导最多三个初始上下文澄清问题（无预制目录）。必须满足：
   - 基于用户措辞 + 从 spec/plan/tasks 中提取的信号生成
   - 仅询问对清单内容有实质性影响的信息
   - 如果已在 `$ARGUMENTS` 中明确，则逐个跳过
   - 优先精准而非全面

   生成算法：
   1. 提取信号：功能领域关键词（如 auth、latency、UX、API），风险指示词（"critical"、"must"、合规性），利益相关者提示（"QA"、"review"、"安全团队"），以及明确的交付物（"a11y"、"rollback"、"contracts"）。
   2. 将信号聚类为候选重点领域（最多 4 个）并按相关性排序。
   3. 在不明确的情况下识别可能的受众和时机（作者、审阅者、QA、发布）。
   4. 检测缺失维度：范围广度、深度/严格性、风险侧重、排除边界、可衡量的验收标准。
   5. 从以下原型中选择问题：
      - 范围细化（如"是否应包含与 X 和 Y 的集成接触点，还是仅限本地模块正确性？"）
      - 风险优先级（如"哪些潜在风险领域应获得强制性门控检查？"）
      - 深度校准（如"这是轻量级的 pre-commit 健全性检查列表还是正式的发布门控？"）
      - 受众框架（如"这是仅供作者使用还是在 PR 审阅期间供同行使用？"）
      - 边界排除（如"这轮是否明确排除性能调优项目？"）
      - 场景类别缺口（如"未检测到恢复流程——rollback/部分失败路径是否在范围内？"）

   问题格式规则：
   - 如果提供选项，生成紧凑表格，列：选项 | 候选 | 为何重要
   - 最多提供 A–E 选项；如果自由形式回答更清晰则省略表格
   - 永远不要让用户复述他们已经说过的话
   - 避免推测性类别（不 hallucinate）。如有不确定，明确询问："确认 X 是否属于范围。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：如果涉及代码则为审阅者（PR）；否则为作者
   - 重点：相关性最高的前 2 个聚类

   输出问题（标记 Q1/Q2/Q3）。收到回答后：如果 ≥2 个场景类别（Alternate / Exception / Recovery / Non-Functional 领域）仍不明确，可以提出最多两个有针对性的后续问题（Q4/Q5），每个问题附一行理由（如"Unresolved recovery path risk"）。总计不超过五个问题。如果用户明确拒绝更多提问，跳过升级。

3. **理解用户请求**: 结合 `$ARGUMENTS` + 澄清回答：
   - 推导清单主题（如 security、review、deploy、ux）
   - 整合用户明确提到的必须包含项
   - 将重点选择映射到类别框架
   - 从 spec/plan/tasks 推断任何缺失的上下文（不要 hallucinate）

4. **加载功能上下文**: 从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围
   - plan.md（如果存在）：技术细节、依赖关系
   - tasks.md（如果存在）：实施任务

   **上下文加载策略**：
   - 仅加载与活跃重点领域相关的必要部分（避免完整文件倾倒）
   - 对于长段落，优先总结为简洁的场景/需求要点
   - 使用渐进式披露：仅在检测到缺口时才进行后续检索
   - 如果源文档较大，生成临时总结项而非嵌入原始文本

5. **生成清单** - 创建"需求单元测试"：
   - 如果 `checklists/` 目录不存在则创建
   - 生成唯一清单文件名：
     - 使用基于领域的简短描述性名称（如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
   - 文件处理行为：
     - 如果文件不存在：创建新文件，条目从 CHK001 开始编号
     - 如果文件存在：将新条目追加到现有文件，从上一个 CHK ID 继续（如上一条为 CHK015，新条目从 CHK016 开始）
   - 永远不要删除或替换现有清单内容——始终保留和追加

   **核心原则 - 测试需求，而非实现**：
   每个清单条目必须评估需求本身的质量：
   - **完整性**：所有必要的需求是否都存在？
   - **清晰性**：需求是否明确且具体？
   - **一致性**：需求是否相互一致？
   - **可衡量性**：需求能否被客观验证？
   - **覆盖率**：是否涵盖所有场景/边缘情况？

   **类别结构** - 按需求质量维度分组：
   - **需求完整性**（是否所有必要的需求都已记录？）
   - **需求清晰性**（需求是否明确且无歧义？）
   - **需求一致性**（需求是否相互协调无冲突？）
   - **验收标准质量**（成功标准是否可衡量？）
   - **场景覆盖率**（是否涵盖所有流程/用例？）
   - **边缘情况覆盖率**（是否定义了边界条件？）
   - **非功能性需求**（性能、安全、无障碍等——是否已指定？）
   - **依赖与假设**（是否已记录并验证？）
   - **歧义与冲突**（需要澄清什么？）

   **如何编写清单条目 - "需求的单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证落地页显示 3 个剧集卡片"
   - "测试桌面端的悬停状态"
   - "确认点击 Logo 导航到首页"

   ✅ **正确**（测试需求质量）：
   - "是否指定了精选剧集的确切数量和布局？" [完整性]
   - "'突出展示'是否用具体尺寸/位置进行了量化？" [清晰性]
   - "所有交互元素的悬停状态需求是否一致？" [一致性]
   - "是否定义了所有交互式 UI 的键盘导航需求？" [覆盖率]
   - "是否指定了 Logo 图片加载失败时的后备行为？" [边缘情况]
   - "是否定义了异步剧集数据的加载状态？" [完整性]
   - "规范是否定义了竞争 UI 元素的视觉层级？" [清晰性]

   **条目结构**：
   每个条目应遵循此模式：
   - 以问题格式询问需求质量
   - 关注规范/计划中**写明**（或未写明）的内容
   - 在括号中包含质量维度 [Completeness/Clarity/Consistency/etc.]
   - 引用规范章节 `[Spec §X.Y]`（检查现有需求时）
   - 使用 `[Gap]` 标记（检查缺失需求时）

   **各质量维度的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理需求？" [Gap]
   - "是否为所有交互元素指定了无障碍需求？" [完整性]
   - "是否为响应式布局定义了移动端断点需求？" [Gap]

   清晰性：
   - "'快速加载'是否用具体时间阈值进行了量化？" [清晰性, Spec §NFR-2]
   - "'相关剧集'的选择标准是否明确定义？" [清晰性, Spec §FR-5]
   - "'突出'是否用可衡量的视觉属性定义？" [歧义, Spec §FR-4]

   一致性：
   - "所有页面的导航需求是否一致？" [一致性, Spec §FR-10]
   - "落地页和详情页的卡片组件需求是否一致？" [一致性]

   覆盖率：
   - "是否为零状态场景（无剧集）定义了需求？" [覆盖率, 边缘情况]
   - "是否解决了并发用户交互场景？" [覆盖率, Gap]
   - "是否为部分数据加载失败指定了需求？" [覆盖率, 例外流程]

   可衡量性：
   - "视觉层级需求是否可衡量/可测试？" [验收标准, Spec §FR-1]
   - "'均衡的视觉重量'能否被客观验证？" [可衡量性, Spec §FR-2]

   **场景分类与覆盖率**（需求质量焦点）：
   - 检查以下场景类别的需求是否存在：主要、备选、异常/错误、恢复、非功能性场景
   - 对于每个场景类别，问："是否[场景类型]需求完整、清晰且一致？"
   - 如果场景类别缺失："是否[场景类型]需求被有意排除或缺失？" [Gap]
   - 当发生状态变更时包含韧性/回滚："是否为迁移失败定义了回滚需求？" [Gap]

   **可追溯性要求**：
   - 最低要求：≥80% 的条目必须包含至少一个可追溯性引用
   - 每个条目应引用：规范章节 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 如果没有 ID 系统："是否建立了需求和验收标准 ID 方案？" [可追溯性]

   **问题呈现与解决**（需求质量问题）：
   询问关于需求本身的问题：
   - 歧义："'fast' 一词是否用具体指标进行了量化？" [歧义, Spec §NFR-1]
   - 冲突："§FR-10 和 §FR-10a 之间的导航需求是否冲突？" [冲突]
   - 假设："'podcast API 始终可用'的假设是否经过验证？" [假设]
   - 依赖："是否记录了外部 podcast API 的需求？" [依赖, Gap]
   - 缺失定义："'视觉层级'是否用可衡量标准定义？" [Gap]

   **内容整合**：
   - 软上限：如果候选条目 > 40，按风险/影响优先级排序
   - 合并检查同一需求方面的近似重复项
   - 如果 > 5 个低影响的边缘情况，创建一个条目："需求中是否解决了边缘情况 X、Y、Z？" [覆盖率]

   **🚫 绝对禁止** - 这些使清单变成实现测试，而非需求测试：
   - ❌ 任何以"验证"、"测试"、"确认"、"检查" + 实现行为开头的条目
   - ❌ 引用代码执行、用户操作、系统行为
   - ❌ "正确显示"、"正常工作"、"按预期运行"
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 流程
   - ❌ 实现细节（框架、API、算法）

   **✅ 必须的模式** - 这些测试需求质量：
   - ✅ "是否为 [场景] 定义/指定/记录了 [需求类型]？"
   - ✅ "[模糊术语] 是否用具体标准进行了量化/澄清？"
   - ✅ "[章节 A] 和 [章节 B] 之间的需求是否一致？"
   - ✅ "[需求] 能否被客观测量/验证？"
   - ✅ "需求中是否解决了 [边缘情况/场景]？"
   - ✅ "规范中是否定义了 [缺失方面]？"

6. **结构参考**: 按照 `templates/checklist-template.md` 中的规范模板生成清单，包括标题、元数据部分、类别标题和 ID 格式。如果模板不可用，则使用：H1 标题、purpose/created 元数据行、`##` 类别部分，包含 `- [ ] CHK### <需求条目>` 行，ID 从 CHK001 开始全局递增。

7. **报告**: 输出清单文件的完整路径、条目数量，并总结是创建了新文件还是追加到现有文件。总结：
   - 选择的重点领域
   - 深度级别
   - 角色/时机
   - 纳入的任何用户明确指定的必须包含项

**重要**: 每次 `__SPECKIT_COMMAND_CHECKLIST__` 命令调用使用简短描述性清单文件名，且要么创建新文件要么追加到现有文件。这允许：

- 多种类型的清单（如 `ux.md`、`test.md`、`security.md`）
- 简单易记的文件名，指示清单目的
- 在 `checklists/` 文件夹中易于识别和导航

为避免混乱，使用描述性类型并在完成后清理过时的清单。

## 反面示例：不应该做什么

**❌ 错误 - 这些测试的是实现，而非需求：**

```markdown
- [ ] CHK001 - Verify landing page displays 3 episode cards [Spec §FR-001]
- [ ] CHK002 - Test hover states work correctly on desktop [Spec §FR-003]
- [ ] CHK003 - Confirm logo click navigates to home page [Spec §FR-010]
- [ ] CHK004 - Check that related episodes section shows 3-5 items [Spec §FR-005]
```

**✅ 正确 - 测试需求质量：**

```markdown
- [ ] CHK001 - Are the number and layout of featured episodes explicitly specified? [Completeness, Spec §FR-001]
- [ ] CHK002 - Are hover state requirements consistently defined for all interactive elements? [Consistency, Spec §FR-003]
- [ ] CHK003 - Are navigation requirements clear for all clickable brand elements? [Clarity, Spec §FR-010]
- [ ] CHK004 - Is the selection criteria for related episodes documented? [Gap, Spec §FR-005]
- [ ] CHK005 - Are loading state requirements defined for asynchronous episode data? [Gap]
- [ ] CHK006 - Can "visual hierarchy" requirements be objectively measured? [Measurability, Spec §FR-001]
```

**关键差异：**

- 错误理解：测试系统是否正常工作
- 正确理解：测试需求是否编写正确
- 错误理解：行为验证
- 正确理解：需求质量验证
- 错误理解："它是否做 X？"
- 正确理解："X 是否被明确指定？"

## 执行后检查

**检查扩展钩子（检查清单生成后）**：
检查项目根目录是否存在 `.specify/extensions.yml`。
- 如果存在，读取文件并查找 `hooks.after_checklist` 键下的条目
- 如果 YAML 无法解析或无效，静默跳过钩子检查并继续正常流程
- 过滤掉 `enabled` 明确设置为 `false` 的钩子。没有 `enabled` 字段的钩子默认视为已启用。
- 对于每个剩余的钩子，**不要**尝试解释或求值钩子的 `condition` 表达式：
  - 如果钩子没有 `condition` 字段，或为 null/空，则将该钩子视为可执行
  - 如果钩子定义了非空的 `condition`，则跳过该钩子，将条件求值留给 HookExecutor 实现
- 对于每个可执行的钩子，根据其 `optional` 标志输出以下内容：
  - **可选钩子**（`optional: true`）：

```
## Extension Hooks

    **Optional Hook**: {extension}
    Command: `/{command}`
    Description: {description}

    Prompt: {prompt}
    To execute: `/{command}`
```

- **强制钩子** (`optional: false`):

```
## Extension Hooks

    **Automatic Hook**: {extension}
    Executing: `/{command}`
    EXECUTE_COMMAND: {command}
```

- 如果没有注册任何 hooks，或 `.specify/extensions.yml` 文件不存在，则静默跳过
